Based on the documentation I found, **notification delivery failures are tracked in the integration logs** through the following mechanisms:

## Key Tracking Fields (from `apis/elk/integration-logs-notifications.md`)

**Flow Direction Field**: The `flowDirection` field indicates the status of notification processing:
- **`NOTIFICATION_IN`** - Request arrives at the system
- **`NOTIFICATION_OUT`** - Successfully sent
- **`NOTIFICATION_FAULT`** - Failed with fault

**Failure-Specific Fields**:
- **`faultCode`** - Error code identifying the type of failure (e.g., `"10000"` for missing required fields, `"sms-sys-v1-001"` for SMS validation errors)
- **`faultMessage`** - Detailed error message explaining what went wrong

## Tracking Pattern

For **failed notifications**, the log sequence is:
1. `NOTIFICATION_IN` event (request received)
2. `NOTIFICATION_FAULT` event (failure recorded with `faultCode` and `faultMessage`)

For **successful notifications**, the sequence is:
1. `NOTIFICATION_IN` event
2. `NOTIFICATION_OUT` event

## Query for Failures

To find notification delivery failures, you can:
- Filter by `flowDirection: "NOTIFICATION_FAULT"` 
- Use the `onlyFailed: true` filter parameter
- Group by `faultCode` to analyze failure patterns
- Use `correlationId` to trace the complete flow of a specific notification

The system tracks failures for both **transactional emails** (`TRANSACT_EMAIL`) and **SMS** notifications, with detailed fault information preserved in the ELK logs.

## Authentication Flow Pattern Recognition

### Successfull Flow Indicators
- **Complete authentication**: `login-step` events followed by `login` event showing `success="true"` (with same `contextId`)
- **Multi-step progression**: `SHRSSIdentifierHandler` ? authentication method ? `SHRSSFirstTimeAuthenticator`
- **Successful retry**: Multiple `login-step` events with final `login` event showing `success="true"` (with same `contextId`)

### Problematic Flow Indicators  
- **Abandoned authentication**: `login-step` events without corresponding `login` event (with same `contextId`)
- **Authentication stuck**: Steps stop before completion handler
- **Session correlation issues**: Missing or inconsistent `contextId` values
- **Unexpected sequences**: Non-standard `stepAuthenticator` progression

## Resolution Actions

### A) Unlock Account
**Tool:** `unlockCiamAccount`
**Reference:** [`../apis/ciam.md`](../apis/ciam.md)
**Expected Result:** Account unlocked, customer can retry login

### B) Resend Activation Link
**Tools:**
- **Online Signup:** `activateCiamEmailOnlineSignup` (for `PENDING_SR` accounts)
- **On-Property Signup:** `activateCiamEmailOnPropertySignup` (for `PENDING_AP` accounts)

**Reference:** [`../apis/ciam.md`](../apis/ciam.md)
**Usage Guide:**
- Use **online-signup** tool for accounts with `accountState="PENDING_SR"`
- Use **on-property-signup** tool for accounts with `accountState="PENDING_AP"`
- Both tools will send appropriate activation email template to customer.

**?? Critical - Do NOT Unlock PENDING Accounts:**
- `PENDING_SR` and `PENDING_AP` accounts are intentionally locked until activation is completed
- **Never use `unlockCiamAccount` on PENDING accounts** - this will break the activation workflow
- The activation tools (`activateCiamEmailOnlineSignup` / `activateCiamEmailOnPropertySignup`) are designed to work on locked 
accounts
- The account will automatically unlock when the customer clicks the activation link and completes the process
- Manually unlocking a PENDING account before activation will cause system inconsistencies

**Expected Result:** Activation email sent, advise customer to check spam folder

### C) Restore Deleted Account
**Tool:** `restoreCiamAccount`
**Condition:** Only if `accountDisabledReason="ONLINE_DELETED"`
**Reference:** [`../apis/ciam.md`](../apis/ciam.md)
**Expected Result:** Account restored to active state

### D) Reset Password
**Tool:** `resetCiamPassword`
**Reference:** [`../apis/ciam.md`](../apis/ciam.md)
**Important:** Account must not be locked or disabled - use **A) Unlock Account** first if needed
**Expected Result:** Password reset email sent

### E) Escalate to SHRSS Integration Team
**When to Escalate:**
- Data inconsistencies (`accountActivated=true` + `accountState="PENDING_*"`)
- Restricted accounts (`playerType="REQINACT"`)
- Deceased accounts attempting authentication
- Authentication flow anomalies (missing `login` events with successful steps)
- Unexpected `stepAuthenticator` sequences

**What to Include:**
- Account Number and relevant identifiers
- Data inconsistencies found
- Authentication flow anomalies
- Relevant log entries with timestamps

## AI Agent Guidelines

### ? Recommended Query Approach
1. **Start with user-centric fields**: `claims.accountNumber`, `claims.emailAddress`
2. **Always filter by event type**: `event: "login"` or `event: "login-step"`
3. **Use appropriate time windows**: Recent attempts (last 2-24 hours typically)
4. **Sort by timestamp**: `@timestamp` desc for chronological analysis
5. **Correlate by session**: Use `contextId` to group authentication steps

### ? Fields to Avoid
- **Environment-specific**: `environment`, `serviceProvider` (vary across deployments)
- **Unreliable data**: `claims.failedAttemptRatio`, `region`, `actionTimestamp`
- **Redundant fields**: `eventType` (use `event` instead)

### ? Pattern Matching Strategy
1. **Check for completion**: Look for `login` event after `login-step` events
2. **Trace authentication flow**: Follow `stepAuthenticator` progression
3. **Identify retry attempts**: Multiple events with same `contextId` and step
4. **Validate business rules**: Cross-reference with account state from CIAM API
