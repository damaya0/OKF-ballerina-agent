**Parameters:**

```
  targetSystem: "all"

    primaryEmail:
      email: "newemail@example.com"
      verified: 0
      status: "S"
  correlationId: "your-correlation-id"
```

1. **Asynchronous Processing**: The update is queued immediately and processed in the background. It may take several minutes (2-5 minutes) to propagate across all systems.

   - `verified: 0` = Email requires verification (customer needs to verify)
   - `verified: 1` = Email has been verified
   - `status: "S"` = Subscribed to marketing
   - `status: "U"` = Unsubscribed
   - `status: "B"` = Bounced

Based on the documentation from `/apis/elk/ciam-logs.md`, here's what the CIAM logs contain:

## Events Captured

CIAM logs capture **authentication events** with two main types:

1. **`login-step` events** ? Individual authentication steps (username validation, password entry, OTP verification, etc.)
2. **`login` events** ? Final authentication result (only appears when all steps eventually succeed)

These logs help explain *why a login succeeded or failed* and track **multi-step** and **multi-factor** authentication flows.

## Available Fields

### Essential Fields:
- **`@timestamp`** - Event time (datetime UTC)
- **`event`** - Event type (`login` or `login-step`)
- **`success`** - Outcome as string (`"true"` or `"false"`)
- **`contextId`** - Session identifier linking all steps in an authentication flow
- **`authenticationStep`** - Step number in the flow (`"0"`, `"1"`, `"2"`, etc.)
- **`stepAuthenticator`** - Authentication handler type (e.g., `SHRSSIdentifierHandler`, `SMSOTP`, `EmailOTP`, `SHRSSFirstTimeAuthenticator`)

### User Identification Fields:
- **`claims.accountNumber`** - Account number (primary identifier)
- **`claims.emailAddress`** - User email
- **`claims.userId`** - Internal user UUID
- **`username`** - CIAM username/UUID

### Authentication Context:
- **`claims.failedReason`** - Human-readable failure explanation (e.g., `"Invalid Credentials"`, `"Account is Locked"`, `"PENDING_SR"`, `"USER_NOT_FOUND"`)
- **`isFirstLogin`** - Whether this was the first successful login
- **`serviceProvider`** - Client/app initiating the login

### Technical Details:
- **`claims.browserAgent`** - Full user agent string
- **`claims.browser`** / **`claims.browserVersion`** - Browser family and version
- **`claims.ipAddress`** - Client IP address
- **`claims.operatingSystem`** / **`claims.osVersion`** - OS family and version

The logs use the ELK index `integration-analytics-ciam` and should always be filtered by `event: "login"` or `event: "login-step"` to get authentication-specific events.

### **Step 1: Gather Information**
- Get the customer's **Account Number** (preferred) or **Email Address**

  - Deceased accounts

  - Data inconsistencies
